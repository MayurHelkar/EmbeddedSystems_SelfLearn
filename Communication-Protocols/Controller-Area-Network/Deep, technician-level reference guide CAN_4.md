# 1. AUTOSAR PDU Router (PduR) – ARXML Configuration Structure

The PDU Router is the **central routing hub** between COM, CAN Interface, and other bus stacks.

---

## 1.1 PDU Router architecture mapping

| Layer  | Function                |
| ------ | ----------------------- |
| COM    | Signal packaging        |
| PduR   | Routing decision engine |
| CanIf  | CAN abstraction         |
| CanDrv | Hardware control        |

---

## 1.2 ARXML structure (simplified OEM representation)

```xml id="pdu_arxml"
<PDU-ROUTER>
  <PDU-ROUTING-TABLE>
    <ROUTE>
      <SOURCE-PDU>ComTxPdu_EngineSpeed</SOURCE-PDU>
      <DESTINATION-PDU>CanIfTxPdu_0x1F0</DESTINATION-PDU>
    </ROUTE>
  </PDU-ROUTING-TABLE>
</PDU-ROUTER>
```

---

## 1.3 Routing configuration breakdown

| XML Element     | Meaning                     |
| --------------- | --------------------------- |
| SOURCE-PDU      | Origin COM signal container |
| DESTINATION-PDU | CAN frame mapping           |
| ROUTE           | routing rule                |
| PDU-ID          | unique message identifier   |

---

## 1.4 Multi-bus routing example

| Source     | Destination    | Action           |
| ---------- | -------------- | ---------------- |
| COM        | CAN Powertrain | forward          |
| CAN Body   | CAN Gateway    | filter + forward |
| Diagnostic | UDS stack      | route to ECU     |

---

## 1.5 PDU routing logic (runtime)

```c id="pdur_logic"
if(pdu.source == COM)
    route_to(CAN_IF);

else if(pdu.source == DIAG)
    route_to_UDS_STACK();

else
    drop();
```

# 2. UDS Security (Seed-Key “Theory Only” – Defensive View)

UDS security access protects ECUs from unauthorized diagnostic control.

---

## 2.1 Security access flow

| Step | Service | Description   |
| ---- | ------- | ------------- |
| 1    | 0x27 01 | Request seed  |
| 2    | ECU     | Sends seed    |
| 3    | Tester  | Computes key  |
| 4    | 0x27 02 | Sends key     |
| 5    | ECU     | Grants access |

---

## 2.2 Seed-key model (conceptual)

| Component | Role                            |
| --------- | ------------------------------- |
| Seed      | challenge from ECU              |
| Key       | response generated by algorithm |
| Algorithm | OEM proprietary function        |
| HSM       | secure key storage              |

---

## 2.3 Security algorithm types (high level)

| Type      | Description               |
| --------- | ------------------------- |
| XOR-based | legacy ECUs               |
| LFSR      | pseudo-random shift logic |
| AES-based | modern ECUs               |
| HSM-bound | hardware cryptography     |

---

## 2.4 Why “cracking” is hard in modern ECUs

| Protection          | Effect             |
| ------------------- | ------------------ |
| Rolling seeds       | prevents replay    |
| Time-bound response | limits brute force |
| HSM computation     | no software access |
| Key diversification | ECU-specific keys  |

---

## 2.5 Defensive analysis model

| Risk                | Mitigation          |
| ------------------- | ------------------- |
| Replay attack       | session binding     |
| brute force         | lockout counters    |
| reverse engineering | firmware encryption |

# 5. EV Architecture: CAN-FD vs Automotive Ethernet Migration

---

## 5.1 Traditional CAN architecture

| Feature     | CAN      |
| ----------- | -------- |
| Speed       | 500 kbps |
| Payload     | 8 bytes  |
| Determinism | High     |
| Cost        | Low      |

---

## 5.2 CAN-FD (Flexible Data Rate)

| Feature             | CAN-FD       |
| ------------------- | ------------ |
| Speed               | up to 8 Mbps |
| Payload             | 64 bytes     |
| Backward compatible | Yes          |
| Use case            | modern ECUs  |

---

## 5.3 Automotive Ethernet architecture

| Feature  | Ethernet           |
| -------- | ------------------ |
| Speed    | 100 Mbps – 10 Gbps |
| Protocol | TCP/IP, SOME/IP    |
| Topology | star               |
| Use case | ADAS, infotainment |

---

## 5.4 EV domain architecture (modern vehicles)

```id="evarch"
[ADAS Ethernet Backbone]
        |
[Central Gateway ECU]
   |        |        |
CAN-FD   CAN-FD   CAN-FD
Power    Chassis   Body
```

---

## 5.5 Comparison table

| Feature  | CAN           | CAN-FD        | Ethernet        |
| -------- | ------------- | ------------- | --------------- |
| Speed    | Low           | Medium        | Very High       |
| Latency  | deterministic | deterministic | variable        |
| Payload  | 8 bytes       | 64 bytes      | large packets   |
| Cost     | lowest        | medium        | highest         |
| Use case | legacy ECUs   | modern ECUs   | ADAS/EV compute |

---

## 5.6 Migration trend in EVs

| Domain          | Migration direction |
| --------------- | ------------------- |
| Powertrain      | CAN → CAN-FD        |
| Body control    | CAN → CAN-FD        |
| ADAS            | CAN → Ethernet      |
| Infotainment    | Ethernet dominant   |
| Central compute | Ethernet backbone   |

---

## 5.7 Real EV architecture trend (Tesla-style model)

| Layer            | Technology          |
| ---------------- | ------------------- |
| Central computer | Ethernet            |
| Vehicle subnet   | CAN-FD              |
| Sensor fusion    | high-speed Ethernet |
| Actuators        | CAN/CAN-FD          |
